Project One

Code:

    Musical Clock


    Video:




    As mentioned earlier, for my P1 Project, I was planning to use music/noise to measure time. It does not follow the standard metric system of hours, minutes and seconds, but instead, finds a new way to measure the time passed based on the amplitude, frequency and BPM (beats per minute) of the songs.

    Strongly inspired by the old Windows Media Player, although we can’t tangibly touch music, I’ve always been fascinated with how we can “see” music through frequency and amplitude visualisation. The original idea was to use p5.js and a microphone as an input to any playlist of the listener’s choice, and the code would generate time passed based on the beats of the song. A slow song would generate slower beat analysis and vice versa.

    However, the constraints of p5.js shaped the project differently.

    So, the first step of generating the code was to create a microphone input (as experimented in week four). The mic.Input function detects only amplitude (in English, it’s called volume. I did not sign up for physics but ended up learning it anyway). Amplitude generates only one output value, which is the current volume the mic detects.

    I continued experimenting with this in week five, and I quickly realised the mic is extremely sensitive, making it difficult to work in DIDI due to the loud air conditioners. I got the best results in a quiet environment. By this point, I realised “beats” are detected by p5.js only at a certain frequency (in English, we call this pitch – how shrill or deep a sound is). Thus, I had to enter the world of p5.FFT.

    FFT analyses the sound input and produces an array of numbers as an output – providing the amplitude of the frequencies it can detect. We can choose to see the number of frequencies p5.js analyses – any number between 2 to 1024, as long as the number is a power of two (e.g. you can only analyse 2, 4, 8, 16, 32, 64, 128, 516, etc. frequencies). I still don’t understand the physics behind the reason these numbers are permitted to use, but I thought it was important to point out.

    Finally, in week six, I managed to get the FFT code to work. Since the ambient noise and the mic sensitivity didn’t allow the code to detect beats properly, I decided to look up and use the exact frequency of drums as the minimum and maximum frequencies to be detected. To my dismay, the ambient sounds (such as air conditioners) unfortunately have similar frequencies, therefore the code worked even without music. I would have preferred creating an audio source within the computer, unfortunately the attempt led to more problems such as having to set up a server to support uploading an mp3 file.

    A certain code called “peakDetect” wasn’t the best either, as it only worked with a certain genre of music, however, it created a decent effect, and I decided to integrate it into my code.

    After a lot of trial and error, I ended up with a rotating pattern that changed its shape based on the volume of the beats/lyrics (whichever was produced to be louder in the song, e.g. harmonies will not trigger the beat but its drums would, and on the other hand a solo singer will overpower the drums in the background). The clock’s pattern would move anti-clockwise slowly if there is no song playing, encouraging you to charge the clock by playing music. Upbeat songs would turn the pattern clockwise more than a slow song would, creating a feeling of fighting the clock’s anticlockwise movement by playing high energy songs.

    I love the how the code works, and I thoroughly enjoyed the process of creating it!
